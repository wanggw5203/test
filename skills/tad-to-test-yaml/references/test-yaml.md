# YAML 测试用例规则

## 结构选择

先检查目标模块附近或项目已约定测试产物目录中的 YAML 示例。格式稳定时沿用本地结构。测试目录优先采用用户指定值；未指定时根据项目现有目录和同类产物确定，不预设固定目录名。

输出名严格采用来源 Markdown 的基础文件名，仅把 `.md` 改为 `.yaml`。同名 YAML 已存在时覆盖该文件；不得另建简称、提炼名、`-new`、副本或序号文件。其他 YAML 只用于参考结构，不得影响当前输出命名。

没有明确本地格式时使用：

```yaml
TestCases:
  - name: ""
    caseId: ""
    level: P0
    before: ""
    module: ""
    tags: 前端
    thought: ""
    steps:
      - step: ""
        expected: ""
    state: valid
```

部分知识库文件没有 `caseId`。目标工具接受时保留该字段；只有本地规范稳定省略时才省略。

## 设计方法映射

- 表格 -> 数据驱动、等价类、边界值。
- 流程图 -> 路径覆盖、场景法。
- 状态描述 -> 状态迁移。
- 条件规则 -> 决策表。
- 明确列出的枚举、节点和状态 -> 逐项覆盖；优先级按业务影响确定，不因“明确列出”自动设为 P0。
- STD 依赖 -> 接口、契约、链路、幂等、重试、降级、稳定性和可观测性。

生成最小但完整的集合，避免用不同措辞重复同一断言。每条核心流程至少保留一个端到端 P0 主路径，再用原子用例覆盖关键分支和边界。

## 优先级规则

- `P0`：核心端到端主流程、阻断规则、必需业务契约、关键状态迁移、核心上下游契约。
- `P1`：重要边界、异常、逆向路径、依赖失败、数据一致性、幂等、权限或数据范围风险。
- `P2`：低风险界面行为、次要交互、兼容性、易用性和无直接阻断影响的稳定性检查。

## 字段规则

- `name`：用中文描述条件和预期结果。
- `caseId`：使用目标仓库约定，保持唯一和稳定。
- `before`：写清环境、数据和状态，保证可执行。
- `module`：使用 `功能用例/模块/子模块/测试类型` 形式，或沿用本地层级。
- `tags`：按实际操作与断言对象选择。直接调用服务并断言返回数据、过滤、统计或状态使用 `服务端`；通过页面操作并断言交互、展示、跳转或提示使用 `前端`；有协议证据且验证请求响应契约时使用 `接口`；有明确 STD 稳定性证据时使用 `稳定性`。
- `thought`：引用来源规则、章节或技术证据，例如 `来源 TRD §2.2-3~5`。
- `steps.step`：一个操作或刺激。
- `steps.expected`：具体可观察结果。
- `state`：可执行用例使用 `valid`；只有本地结构支持时才使用废弃或草稿状态。

## 生成规则

- 一个用例验证一个核心断言。
- 端到端主路径是例外：可以包含多个连续步骤，但必须属于同一条业务路径，并从前置条件走到最终业务结果；不能在一个主路径用例中混入对称入口或无关分支。
- 必填校验原则上按字段拆分；本地测试策略明确要求组合校验时除外。
- 不同入口、角色、路由条件、数据范围和依赖模式分别生成用例。
- 没有 STD 证据时不生成技术实现断言。
- STD 仅写“未发现明确信息”或“无法获取链路”时，禁止生成接口、数据库、缓存、定时任务、幂等、重试或降级用例。
- 不得只写“页面正常”；应写明展示、变化、阻断、创建、导出或同步结果。
- 精确保留状态、提示语、枚举标签和阈值。
- 唯一性规则在 `before` 中说明动态唯一数据的准备方式。
- 变量类规则按变量拆分，明确输入值、来源和最终替换结果；原文存在空值规则时另生成空值用例。
- 数据驱动用例在 `before`、步骤或同一用例的数据块中嵌入完整枚举与预期映射，不允许使用“参考 TAD”“按配置逐项”作为唯一测试数据。
- 事件类规则同时覆盖目标事件和非目标事件，后者应明确断言不产生副作用。
- 示例与预期逻辑冲突时，保留原始示例并在 `thought` 或待确认项中标记冲突。
- 同一数据规则需要验证接口返回和页面呈现时拆成服务端、前端两条用例；没有端点、服务能力或调用方式证据时，不虚构服务端步骤。
- `before` 从当前需求提取环境、主体、资格或权限、配置、输入、时间、状态、依赖和初始结果等相关维度，不预设业务实体；成组数据使用中性标识并明确唯一差异字段。
- 步骤写明入口、筛选值、输入值、触发动作和观察位置；预期逐步写明具体对象、字段、数量、状态、文案或副作用，不使用通用结论代替需求规则。

## 校验清单

- YAML 能成功解析。
- `TestCases` 是列表，每条用例包含 `name`、`level`、`before`、`module`、`tags`、`thought`、`steps` 和 `state`。
- 每个步骤包含 `step` 和 `expected`。
- 用例名唯一；存在 `caseId` 时其值唯一。
- P0 覆盖全部主流程及原文明确列出的枚举、节点和状态。
- 每条 `thought` 都指向 TAD、TRD 或 STD 来源。
- 不包含无依据的路由、字段、提示语、接口或数据库结论。
- P0 用例能够串起每条核心流程的前置、动作和最终结果；原子用例能够定位关键判断失败。
- 每个非功能用例的 `thought` 指向 STD 中可定位的具体证据，而不是泛称“依据 STD”。
- 完整文件不存在零宽字符、不可见控制字符或根节点外游离文本。
- `tags` 与实际验证对象一致，不存在页面操作用例误标服务端，或无接口证据却虚构服务端调用。
- 每条用例的造数、步骤和预期能够脱离 TAD 单独执行，且仍保留规则来源和证据边界。
- 输出 YAML 与来源 Markdown 基础文件名完全一致；同名旧文件被覆盖，且没有产生任何衍生命名文件。
- 输出目录来自用户指定或项目现有测试产物约定，没有套用与当前项目无关的固定目录映射。
