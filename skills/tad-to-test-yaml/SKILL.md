# TAD 转 YAML 测试用例

把分析文档中的规则转换为最小但完整的结构化测试集。开始前必须阅读 [YAML 测试用例规则](references/test-yaml.md)。

## 确认输入与格式

1. 读取 TAD、TRD、STD 和待确认项。
2. 在生成内容前确定唯一目标路径：目标文件名必须与来源 Markdown 的基础文件名完全一致，只把 `.md` 替换为 `.yaml`；输出目录使用用户指定目录，未指定时识别项目现有测试产物目录约定。
3. 目标路径已存在时覆盖同名 YAML；不得创建简称、提炼名、`-new`、副本、序号或其他衍生文件。
4. 查找目标模块附近或项目测试产物目录中的 YAML，优先沿用本地字段和结构习惯，但不得沿用其他文件的名称。
5. 没有稳定本地格式时，使用参考文件中的标准 `TestCases` 结构。
6. 记录分析规则编号，作为每条用例 `thought` 的追溯来源。

## 设计用例

按分析材料选择测试设计方法：

- 表格：数据驱动、等价类、边界值。
- 流程图：路径覆盖和场景法。
- 状态描述：状态迁移。
- 条件组合：决策表。
- 明确枚举、节点或状态：逐项覆盖。
- STD 技术依赖：接口契约、链路、幂等、重试、降级、稳定性和可观测性。

一个用例只验证一个核心断言。不同入口、角色、路由条件、数据范围、依赖模式和业务结果必须拆分。

用例集同时包含两种层级：

- **端到端主路径**：每条核心流程至少生成一个 P0 场景，从业务前置条件开始，走到最终可观察结果。允许一个用例包含多个连续步骤，但所有步骤共同验证同一条完整业务路径。
- **规则级原子用例**：对关键判断、反向分支、边界、字段和展示分别生成独立用例，便于失败定位。

事件、监听、回调或状态变化类功能必须成对检查：目标事件触发预期动作，非目标变化不触发动作。

## 设置优先级

- `P0`：核心端到端主流程、阻断规则、关键契约、关键状态迁移和核心上下游链路。
- `P1`：重要边界、异常、逆向路径、依赖失败、数据一致性、幂等、权限和数据范围。
- `P2`：低风险界面行为、次要交互、兼容性、易用性和非阻断稳定性检查。

明确枚举不自动等于 P0：影响核心路由、资格、金额、状态或创建结果的枚举为 P0；普通反向分支和重要边界通常为 P1；纯视觉或低风险展示枚举为 P2。

不要为了数量重复表达同一断言。优先生成覆盖高风险规则的最小用例集。

## 生成 YAML

每条用例必须：

- 使用中文描述条件和预期结果。
- 在 `before` 中写清环境、数据和前置状态。
- 在 `thought` 中引用 TAD/TRD/STD 的规则或章节。
- 让每个步骤只包含一个操作或刺激。
- 让每个预期结果可观察，不使用“页面正常”“功能正常”等空泛描述。
- 保留原文中的状态、提示语、枚举标签、阈值和业务术语。
- 对模板变量、占位符和动态字段，分别校验取值来源、替换结果、计算公式及原文明确的空值语义。
- 数据驱动用例必须把枚举值、输入组合和预期映射直接写入 YAML；不得只写“按 TAD 逐项验证”或要求执行者另行查表。

### 判断验证层与标签

按用例实际操作和断言对象设置标签，不按规则看起来属于哪个研发角色猜测：

- `服务端`：直接调用有证据支持的接口或服务能力，断言返回字段、数据范围、过滤、排序、统计、状态或持久化结果。
- `前端`：通过页面完成点击、输入、选择、弹窗、按钮状态、提示、列表展示、跳转或刷新，并断言用户可见结果。
- `接口`：本地规范把协议契约与服务端业务校验分开时使用，重点验证请求参数、响应结构、错误码或上下游契约。
- `稳定性`：仅在 STD 提供明确稳定性目标或实现证据时使用。

接口返回的数据规则可以生成服务端用例，页面如何消费和展示同一数据应另生成前端用例。一个用例同时包含接口断言和页面断言时优先拆分。TAD 未提供端点、服务能力或可调用方式时，不得仅因数据可能来自后端而虚构服务端用例；此时保留可通过页面验证的前端用例，并在 `thought` 中说明证据边界。

### 细化前置、步骤与预期

- `before` 必须从当前需求动态提取造数维度，把抽象条件展开为可识别的环境、主体、权限或资格、配置、输入数据、时间、状态、依赖和初始结果；只写与当前断言有关且有来源依据的字段，不预设特定业务实体。
- 使用成组数据时为每组数据设置中性标识，如“数据A”“场景B”，并写清唯一差异条件及各自预期；标识可按需求实体替换，但不得把其他需求的字段或示例带入当前用例。
- 步骤应从进入需求规定入口或调用有证据的服务能力开始，逐步写明选择的筛选值、输入的需求数据、触发动作和观察位置；不得只写“进入页面”“执行搜索”“查看结果”。
- 每一步预期必须对应本步动作，明确返回或展示的人员、字段、数量、状态、文案、按钮可用性、筛选保留情况或副作用；不得使用“正确展示”“符合预期”“校验成功”等通用表述。
- `thought` 不仅引用章节，还要指出本用例使用的具体规则组合或判定公式；涉及待确认项时明确哪些内容不能形成确定断言。

## 校验结果

1. 使用 YAML 解析器检查语法。
2. 检查根节点、必填字段、步骤和预期结果。
3. 检查用例名称唯一；存在 `caseId` 时检查其唯一性。
4. 检查所有 P0 主流程、明确枚举和关键状态均被覆盖。
5. 检查每条 `thought` 都可回溯到分析来源。
6. 删除没有来源支持的路由、字段、提示语、接口和数据库断言。
7. 非功能或实现级用例必须引用 STD 中明确命名的端点、链路、缓存或稳定性证据；STD 声明证据不足时，不得生成此类用例。
8. 检查完整文件末尾无不可见控制字符、零宽字符或 YAML 根节点之外的游离内容，并对完整文件执行解析，不得只截取可解析片段。

## 输出约束

- 使用用户明确指定的测试目录；未指定时，从项目现有结构中识别测试产物目录。无法确定目录时才请求用户确认，不自行假设 `src/test`、`test` 或其他固定路径。
- 无论目录如何确定，输出 YAML 都必须严格保留来源 Markdown 的基础文件名。
- 同名目标文件存在时直接覆盖；覆盖前确认它正是当前来源 Markdown 映射出的唯一目标。
- 禁止另建简称、优化名、提炼名、`-new`、`-副本`、带序号版本或其他并行用例文件。
- 待确认规则不得伪装成确定用例；应在 `thought` 中标记，或暂不生成可执行状态。
- 只在本地结构明确支持时省略 `caseId` 或使用非 `valid` 状态。
