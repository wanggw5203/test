# 阶段交付契约

## 总原则

- 上一阶段的确定信息成为下一阶段输入，待确认信息不得自动升级为事实。
- 业务规则、页面操作证据和接口框架证据分别管理：业务规则回答“测什么”，页面或框架证据回答“怎么执行”。
- 每个阶段先复用目标模块附近的稳定格式和黄金样例。
- 每个阶段都执行最小必要校验，失败时先诊断和修复；外部阻塞或业务决策不得伪装成通过。

## 阶段输入输出

| 阶段 | 核心输入 | 核心输出 | 必须保留的追溯字段 |
| --- | --- | --- | --- |
| 需求转 TAD | PRD、原型、评审、技术证据 | TAD/TRD/STD Markdown | 需求章节、规则编号 |
| TAD 转 YAML | TAD/TRD/STD、本地 YAML 格式 | 功能与非功能 YAML 用例 | `thought`、`caseId` |
| YAML 转 ATC | 功能 YAML、黄金 ATC、页面证据 | `.atc.yaml` | `functionalCaseIds`、描述 |
| ATC 执行反馈 | ATC、日志、页面证据、执行器配置 | 修复结果、反馈报告、黄金基线 | `runId`、根因、提交 SHA |
| YAML 转接口脚手架 | 接口 YAML、黄金接口测试、API/RO/DTO | Java、suite YAML、JSON5、manifest | `caseId`、`apiCode`、源规格 |
| 接口执行反馈 | 脚手架、日志、HTTP 证据、数据准备器 | 可执行用例、反馈报告、黄金样例 | `runId`、根因、manifest |

## 阶段门禁

### TAD 门禁

- 规则具备条件、动作和可观察结果。
- 明确枚举、状态、阈值、角色和分支未遗漏。
- 技术结论均有证据。
- 冲突和缺失信息已列入待确认项。

### YAML 门禁

- YAML 解析成功。
- 必填字段和步骤结构完整。
- P0 覆盖主流程、阻断规则和明确枚举。
- `thought` 可回溯到 TAD、TRD 或 STD。

### ATC 门禁

- 自动化范围由用户确认；未确认时只覆盖保存主流程。
- 页面路径、标签、提示语和操作顺序有真实证据。
- 动态数据可重复运行，上传资源真实存在。
- 步骤原子化，最终断言与范围一致。
- 项目校验器通过；无法运行时完成静态检查并说明原因。

### 接口脚手架门禁

- 用例标签为 `接口` 或 `服务端`，接口契约与 Java 类型有本地证据。
- 生成规格可解析，初始严格 manifest 校验通过。
- 运行时数据、环境数据、业务枚举、唯一字段和反向非法值已分类。
- 二次加工不删除场景、不缩小覆盖、不放宽期望。
- 测试编译通过；真实环境不可用时明确记录阻塞原因。

### 执行反馈门禁

- 明确失败阶段、根因分类以及被测业务动作是否执行。
- 修复目标与根因一致，没有删除步骤、削弱断言或盲目重试。
- 目标单例复跑通过；页面漂移、等待或瞬时问题连续两次通过后才升级黄金脚本。
- TAD、YAML、ATC 和页面证据的冲突已更新或标记待确认。
- 环境、权限和产品缺陷不写入正确业务期望。
- 接口反馈区分脚手架错误、数据准备失败、实际接口失败和字段断言失败。

## 冲突处理

- 用户确认决定自动化范围。
- 已确认需求或 TAD 决定业务规则。
- 功能 YAML 决定自动化需要验证的断言。
- 黄金 ATC 决定复杂控件的操作方式。
- 真实页面决定标签、占位符、提示语和跳转结果。
- 黄金接口测试决定包结构、DataProvider、API 调用和断言形状。
- 接口文档、RO/DTO、枚举字典和真实响应共同约束请求与期望数据。
- 通过执行且证据完整的结果可以升级黄金基线；单次偶然通过不能覆盖已确认规则。

不同信息类型不能互相替代。例如，原型可以说明字段布局，但不能在没有需求依据时新增业务规则。
