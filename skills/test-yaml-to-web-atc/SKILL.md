# YAML 转 Web ATC

把功能断言转换为可执行的页面操作。开始前必须阅读 [Web ATC 生成规则](references/web-atc.md)。

## 收集生成依据

按优先级收集：

1. 用户确认的自动化范围，例如仅保存、保存并提交或完整审批。
2. TAD、TRD 或原始需求中的业务规则。
3. 功能 YAML 中需要验证的断言。
4. 已成功运行的 `.atc.yaml` 黄金脚本和真实浏览器页面。
5. 截图、`relationPages`、Axure 和现有上传资源。
6. 项目的 `package.json`、`config.yaml` 和执行器约定。

功能 YAML 决定“验证什么”，黄金脚本和页面证据决定“如何操作”。缺少黄金脚本或真实页面证据时，只能生成待校准草稿，并明确标记 `待真实页面校准`。

## 确定生成范围

默认先生成一条稳定的 P0 主流程：

- 用户未确认提交或审批范围时，默认只覆盖保存。
- 成功路径使用合法、动态、唯一且可重复执行的数据。
- 不生成注释掉的批量用例、不引用不存在的文件、不保留占位值。
- 字段边界、异常、权限和上传失败等 P1/P2 场景仅在用户要求时扩展。

## 生成 ATC

1. 沿用目标模块附近 `.atc.yaml` 的 `suite`、`TestCases` 和步骤结构。
2. 使用 `functionalCaseIds` 或描述建立 YAML 功能用例到 ATC 的追溯关系。
3. 一个步骤只执行一个动作或断言。
4. 分开上传、裁剪确认、滚动、保存、等待和最终断言。
5. 优先使用 `input`、`select`、`click`、`upload`、`scroll`、`assert`、`wait` 和 `navigate`。
6. 只有确定性动作无法表达一个复杂交互时才使用 `ai`；一个 `ai` 步骤不得包含多个操作或多个场景。
7. 复用黄金脚本对树形选择器、级联选择器、裁剪弹窗、右键菜单和等待条件的处理方式。

## 约束测试数据

- 有唯一性约束的成功场景使用动态唯一数据。
- 仅允许中文的姓名字段不得拼接数字。
- 可重复成功路径不得使用固定手机号或身份证号。
- 页面校验身份证校验位时使用合法身份证号码。
- 失效日期使用相对日期或未来安全日期。
- 紧急联系人信息与被创建人员不同。
- 固定重复数据只用于重复校验场景。

## 校验与试跑

1. 从项目根目录检查 `package.json` 中的可用脚本。
2. 优先运行项目提供的校验命令，例如 `npm run case:validate`。
3. 执行器支持文件或用例筛选时，只校验新生成的 `.atc.yaml`。
4. 修复 YAML、结构、变量和资源错误。
5. 环境允许且操作安全时，对单条新用例执行 headed 或 debug 试跑。

不能运行执行器时，按参考文件的静态清单检查，并在结果中说明原因。

## 输出约束

- 不虚构路由、字段标签、占位符、提示语、跳转结果和上传文件。
- 页面保存后关闭时，断言成功提示或返回列表中的记录，不断言已经关闭页面内的字段。
- 冲突按“用户范围 > 业务规则 > 功能 YAML > 黄金脚本操作 > 页面证据字段”的信息类型分别处理，并记录未解决冲突。
- 默认把 `src/test/.../功能.yaml` 映射为 `src/test/.../功能.atc.yaml`，同时遵循本地命名方式。
