# ATC 执行反馈闭环

把一次 ATC 执行结果转化为有证据的根因、最小修复、复跑结论和可追溯的知识沉淀。开始前必须阅读 [ATC 失败与修复规则](references/failure-and-repair.md)；输出反馈报告时阅读 [反馈报告结构](references/feedback-report.md)。

## 不可违背的输出协议

- 默认输出必须以 YAML 代码块中的 `feedback:` 根节点开始；Markdown 只能放在 YAML 之后作为补充。
- `feedback.classification` 必须使用本 Skill 定义的 ATC 专用枚举。
- 禁止把 `REQUEST_CONSTRUCTION_ERROR`、`COMPILE_OR_LOAD_ERROR` 等通用测试分类写入主分类；需要保留时只能写入 `diagnosticAlias`。
- 未实施修复或未完成复跑时，不得把状态写为 `passed`，也不得升级黄金脚本。
- 缺失字段使用空值、`unknown` 或待确认项，不得删除结构化字段。
- 诊断证据、修复摘要、复跑结果、上游反馈和阻塞项默认使用中文；分类码、命令、路径和原始错误文本保持原值。

## 收集闭环证据

读取所有可用材料：

- 目标 `.atc.yaml`、关联功能 YAML、TAD/TRD/STD 和原始需求。
- 实际执行命令、控制台输出、校验结果、失败步骤、堆栈和退出码。
- 截图、视频、Trace、浏览器日志和执行前后的页面状态。
- `package.json`、`config.yaml`、执行器版本、环境与用户配置。
- 同页面已跑通的黄金 ATC、真实页面证据、上传资源和相关 Git 差异。

收集全部失败和跳过用例，不得只处理第一条错误。对 Token、Cookie、手机号、身份证号和账号信息脱敏。

## 定位失败阶段

按实际执行链路定位失败发生在哪一层：

```text
解析与结构 -> 环境与认证 -> 前置数据 -> 页面导航
-> 元素定位 -> 页面动作 -> 等待与状态变化 -> 业务断言 -> 清理
```

必须明确被测业务动作是否真正执行。例如，保存按钮未点击成功时，后续“未出现成功提示”只是表象。

## 分类根因

`feedback.classification` 必须从以下枚举中选择一个主分类，不得使用其他 Skill 的分类替代。协同通用诊断 Skill 时，可把其分类记录为辅助别名，但不能覆盖 ATC 主分类：

- `ATC_PARSE_OR_SCHEMA_ERROR`
- `LOCATOR_OR_PAGE_DRIFT`
- `ACTION_SEQUENCE_OR_WAIT_ERROR`
- `TEST_DATA_OR_PREREQUISITE_ERROR`
- `AUTH_OR_PERMISSION_ERROR`
- `ENV_OR_SERVICE_ERROR`
- `ASSERTION_OR_EXPECTATION_ERROR`
- `PRODUCT_BEHAVIOR_ERROR`
- `REQUIREMENT_OR_CASE_CONFLICT`
- `UNKNOWN`

相同根因的失败合并诊断，一次修复共性问题。

例如，目标按钮名称与真实页面及黄金脚本不一致，应归类为 `LOCATOR_OR_PAGE_DRIFT`，不能归类为通用的请求构造错误。

## 实施最小修复

只有正确结果能被项目约定、黄金脚本或真实页面证据证明时才自动修改：

- 修复 YAML 结构、变量引用、资源路径和确定性动作参数。
- 按真实页面修复字段标签、控件定位、滚动、弹窗、等待和动作顺序。
- 将固定或失效成功数据改成合法动态数据。
- 保留功能 YAML 和 TAD 中有证据支持的业务断言。

禁止通过删除步骤、削弱断言、改成跳过、无限等待或盲目重试掩盖失败。业务规则或产品行为冲突时生成待确认反馈，不得自行修改期望。

同一根因最多自动执行一轮“修复 + 复跑”。复跑产生新证据时，重新分类后再决定是否修改。

## 分层复跑

按最小有效范围依次执行：

1. YAML 解析、结构和资源检查。
2. 项目校验命令，例如 `npm run case:validate`。
3. 原失败用例的单例执行。
4. 需要观察页面时执行 headed 或 debug。
5. 只有修改共享黄金步骤、公共变量或执行器配置时才扩大回归范围。

保留原始失败日志和复跑日志。相关用例未通过前，不得宣称闭环完成。

## 回写与沉淀

根据根因选择唯一主回写目标：

- ATC 交互问题：更新 ATC；稳定通过后记录为黄金脚本。
- 测试数据问题：更新变量或造数规则；可复用能力沉淀到数据规则。
- YAML 覆盖或断言问题：只有业务证据充分时更新功能 YAML，并保留 `thought` 追溯。
- 需求冲突：先更新待确认项；确认后按 TAD -> YAML -> ATC 顺序传播。
- 环境、认证或执行器问题：更新环境预检或执行规则，不修改业务期望。
- 产品缺陷：保留失败用例和证据，不把当前错误行为写成正确期望。

黄金脚本不要求新增未经执行器支持的字段。把 ATC 路径、用例名、通过命令、执行器版本、环境、时间和提交 SHA 写入反馈报告；项目已有黄金目录或命名规范时沿用该规范。

## 完成条件

只有同时满足以下条件才标记闭环完成：

- 根因有直接证据，修复目标与根因一致。
- 结构校验和目标单例复跑通过。
- 动态数据和上传资源可再次执行。
- ATC、功能 YAML 与 TAD 不存在未记录冲突。
- 已输出反馈报告，并记录黄金脚本或剩余阻塞。

页面漂移、动作等待和瞬时问题修复后，必须连续两次单例通过再升级为黄金脚本，不能降级为建议项。外部环境、权限或业务决策阻塞时，输出 `blocked`，保留原期望并停止自动修改。

## 与已有 Skill 协同

相关 Skill 可用时按问题类型协同：

- ATC 需要基于黄金脚本重新加工时，使用 `$atc-case-generator` 或 `$test-yaml-to-web-atc`。
- 日志同时涉及编译、HTTP、RPC、服务端或通用测试框架时，使用 `$diagnose-and-repair-tests` 扩展诊断。
- 业务规则确认发生变化时，依次使用 `$requirement-to-tad`、`$tad-to-test-yaml` 和 `$test-yaml-to-web-atc` 传播变更。

协同 Skill 不可用时，仍按本 Skill 的证据、修复、复跑和反馈规则完成可执行部分，并记录缺失能力。
