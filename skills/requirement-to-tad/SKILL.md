# 需求转 TAD 分析

把原始需求提炼成可验证、可追溯、可继续生成测试用例的 TAD 文档。开始前必须完整阅读 [TAD 分析规则](references/tad-analysis.md)。

## 收集上下文

优先收集与目标功能最接近的材料：

- 原始 PRD、飞书内容、Axure 导出、截图、流程图、评审记录和接口说明。
- 同模块 `src/main` 下已有需求或分析文档。
- 同业务域的 TAD、TRD、STD 示例及术语。
- 代码、接口、表结构或配置中的明确技术证据。

按以下优先级确定输出基准：

1. 用户明确提出的结构与表达要求。
2. 目标需求目录下已有的稳定 TAD 模板。
3. 同业务域已有 TAD 的通用表达习惯。
4. 本 skill 的默认结构。

用户提供的期望文档只用于提炼结构、章节层级和表达风格，除非用户明确说明它也是事实来源。不得从样例复制业务结论、接口、Job、cron、表、缓存或调用链。

信息冲突时，保留原文并记录冲突，不得自行修正业务含义。

## 执行分析

1. 识别角色、入口、对象、动作、条件、分支、结果、依赖、异常、数据和状态。
2. 把复合描述拆成原子事实，逐条保留来源位置。
3. 使用“条件 -> 动作 -> 结果”改写为可验证业务规则。
4. 优先用分层编号、条件块和项目列表表达规则；对生命周期对象使用状态迁移，对多分支流程使用 Mermaid。只有当行列对照明显比文字更清晰时才使用表格。
5. 将有技术证据的内容写入 STD；没有证据的内容写为 `未发现明确信息` 或列入待确认项。
6. 合并重复规则，但保留最强约束、精确枚举、阈值、提示语和角色条件。
7. 仅当当前可访问材料中确实包含对应代码或技术文档时，才检索 Job、Dubbo/API、通知记录、配置、缓存和数据库实现并补足 STD。不能访问代码、检索不到实现，或只能在样例 TAD 中看到技术结论时，不得生成链路；STD 明确写明“未获取到可验证的代码或技术证据，无法分析”。
8. 将疑点尽量放在对应业务规则或实现细节附近，以 `> ⚠️ **TODO[REV-n]**` 表达；仅当本地模板使用集中式待确认表时才集中列出。
9. 对多条件、多角色、多状态或多结果规则逐组完整展开。优先使用“场景标题 + 条件 + 动作 + 结果”结构，不用“普通场景”“其他情况”等概括替代原文明确组合。
10. 对源文档中的枚举、映射、精确文案、操作项、提示语和校验条件建立完整性清单；正文可归纳逻辑，但必须保留全部可测试数据。
11. 阅读截图和原型中的可观察信息，记录页面颜色、标题、按钮、显隐、布局和跳转等测试预期，并标明其来源为截图或原型；不得从视觉样例推断技术标识。
12. 扫描空表、空章节、只有标题的规则和未填写单元格，将缺失上下文就近标记为 TODO，不得静默跳过。
13. 业务规则按需求自身的业务能力自然分组，不预设主题名称或数量。每个主题使用“加粗规则标签 + 子项”展开，避免把单一字段、操作项或边界各自升级成独立标题。
14. 当不同入口、角色或模式会产生不同路由时，优先拆成独立流程图。每个判断分支的下一节点必须写明具体动作或目标，跳转后的提示、回流、定位、置顶、刷新等结果使用独立后置节点表达。
15. 业务规则围绕业务机制和决策关系归并，不机械照抄页面控件顺序。路由类规则统一按“选择对象/触发事件 → 动作 → 路由判断 → 页面或业务结果 → 后置表现”展开。

## 生成文档

默认使用以下一级结构；目标仓库已有稳定模板时，必须沿用本地模板：

```markdown
# 业务背景
# TRD 测试需求分析文档
# STD 系统测试分析文档
```

至少包含业务流程、业务规则、依赖关系和技术实现证据。规则合并说明、来源编号与待确认问题的呈现方式服从本地黄金文档，不强制增加本地模板不存在的附录。让每条业务规则都能直接转成一个或多个测试断言。

按独立业务场景拆分 Mermaid 图，业务规则按主题分节。STD 的详细程度由证据决定：有代码证据才列端点和时序；没有证据时保留 STD 标题并说明无法获取链路，不用占位图模拟实现。

## 校验结果

完成前检查：

- 每条规则都有明确条件和可观察结果。
- 每个流程分支、枚举项、状态和边界均被保留。
- 每张配置表的原始行数、枚举数量和输出 TAD 的记录数量一致，或有明确的合并映射说明。
- 每个多维业务场景都覆盖适用对象、条件、动作、结果、涉及数据和可观察结果。
- 每个空表、空章节和缺失单元格均已转成待确认项。
- 已执行表格必要性检查：能用清晰列表表达的内容不使用 Markdown 表格；静态映射很多时也优先按对象分组列出。
- 已执行层级检查：三级标题代表稳定业务主题，四级标题仅用于主题内的独立子流程；不存在只有一两行内容的碎片化标题。
- 依赖项包含失败、超时、重试、降级或补偿语义；原需求未说明时标记待确认。
- Mermaid 与文字规则一致。
- 每个判断节点的所有分支都有明确边标签和具体下一步，不存在“处理结果”“进入对应页面”“按规则执行”等可被继续拆解的概括节点。
- 每条流程终点都包含可观察业务结果；跳转后的提示、回流、定位、排序、刷新或状态变化未被省略。
- 不同入口或模式的结果存在差异时已拆图或明确分区，未通过一张过度复杂的总图隐藏分支。
- 没有虚构接口、字段、路由、数据库、选择器或提示语。
- 待确认项包含上下文、疑点及其对测试设计的影响。
- 文档的标题层级、用词和规则颗粒度与用户要求一致。
- 所有写入 STD 的接口、Job、cron、表名和缓存均能在代码或技术材料中定位；不能定位的内容不得沿用样例猜测。

## 输出约束

- 保留源文档中的中文业务术语。
- 默认写入原始需求文件所在目录；只有用户明确指定其他输出目录时才改变位置。
- 生成文件名前，结合需求标题、背景和核心范围提炼“业务对象 + 核心能力”。删除编号、版本标记以及“需求”“PRD”“方案”“文档”等过程性词；不要机械复制源文件名，也不追加 `-TAD`。
- 不静默覆盖已有文件；覆盖意图不明确时创建新文件或说明冲突。
- 在规则标题或规则正文中保留来源编号，供后续 YAML 用例的 `thought` 字段引用。
