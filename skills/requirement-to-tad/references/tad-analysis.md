# TAD 分析规则

## 输入材料

尽可能收集以下材料：

- PRD、飞书正文、评审记录、原型说明、截图、流程图、接口字段。
- 同模块 `src/main` 下的已有文档。
- 同业务域的 TAD、TRD、STD 示例。
- 代码、API 文档、数据库字段、配置或开发说明中的明确技术线索。

## TAD、TRD 与 STD 的关系

将 TAD 视为总分析文档：

- TRD 聚焦业务规则、业务流程、角色、数据、状态和依赖。
- STD 聚焦有证据支持的技术实现、接口链路、时序、配置、缓存、任务和异常处理。
- 原始材料缺少技术证据时，允许 STD 内容较少，但不得凭经验补全实现。

## 推荐结构

```markdown
# 业务背景
## 1. 背景说明
## 2. 名词解析

---

# TRD 测试需求分析文档
## 1. 业务流程图
## 2. 业务规则
## 3. 业务依赖关系

# STD 系统测试分析文档
## 1. 技术实现
## 2. 时序图
## 3. 实现细节

## 附：规则合并说明
## 附：待确认问题
```

## 原子事实拆解

先把源内容拆成原子事实：

- 入口：页面、菜单、任务、接口、事件、定时任务。
- 角色：用户角色、权限、系统、上游域、下游域。
- 对象：订单、发票、员工、班次、园区、配送单、配置项。
- 动作：新建、编辑、查询、审批、取消、导入、导出、同步。
- 条件：状态、时间、阈值、枚举、灰度开关、权限、上游值。
- 分支：成功、失败、空值、重复、超时、降级、部分成功。
- 结果：页面展示、状态变化、记录生成、提示信息、下游事件。
- 依赖：服务、数据表、配置中心、外部系统、用户权限。
- 失败语义：阻断、重试、忽略、降级、补偿、保留原值。
- 数据规则：必填、唯一、格式、范围、边界、精度。

## 规则标准化

- 使用“条件 -> 动作 -> 结果”表达单条规则。
- 使用决策表表达多个条件组合。
- 使用状态迁移表达有生命周期的业务对象。
- 使用 Mermaid 流程图或时序图表达多分支流程。
- 合并重复规则，但保留最严格的约束。
- 为规则保留来源章节或编号，形成追溯链。

## 业务规则分类

按功能实际情况选择分类：

- 查询、筛选、列表和详情展示。
- 新建、编辑、删除、启用、停用。
- 必填、格式、唯一性、枚举、边界和精度。
- 条件展示、隐藏、禁用和自动回填。
- 导入、上传、导出和下载。
- 审批、状态迁移、取消和失效。
- 上游回调、下游同步、定时任务和消息事件。
- 权限、数据范围、灰度和功能开关。
- 异常、降级、重试、幂等和并发。

## STD 证据边界

只有下列证据存在时才写技术分析：

- 明确命名的接口、DTO、事件、数据表、配置、缓存、定时任务或中间件。
- 需求中明确写出的实现说明。
- 用户提供或仓库内存在的代码与技术文档。

证据不足时写 `未发现明确信息`，或增加待确认项。不得虚构内部服务、字段、选择器、路由、SQL 或提示文案。

## 质量检查

- 每条业务规则至少能生成一个测试用例。
- 每个流程分支都有明确预期结果。
- 原文明确列出的枚举、节点和状态全部保留。
- 阈值、限制、精度、日期范围、提示语、路由条件和角色条件未被改写。
- 依赖关系包含失败语义。
- 图和文字之间不存在矛盾。
- 冲突或可疑示例进入待确认项，不静默修正。
