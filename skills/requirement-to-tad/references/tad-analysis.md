# TAD 分析规则

## 输入材料

尽可能收集以下材料：

- PRD、飞书正文、评审记录、原型说明、截图、流程图、接口字段。
- 同模块 `src/main` 下的已有文档。
- 同业务域的 TAD、TRD、STD 示例。
- 代码、API 文档、数据库字段、配置或开发说明中的明确技术线索。

## TAD、TRD 与 STD 的关系

将 TAD 视为总分析文档：

- TRD 聚焦业务规则、业务流程、角色、数据、状态和依赖。
- STD 聚焦有证据支持的技术实现、接口链路、时序、配置、缓存、任务和异常处理。
- 原始材料缺少技术证据时，允许 STD 内容较少，但不得凭经验补全实现。

## 推荐结构

```markdown
# 业务背景
## 1. 背景说明
## 2. 名词解析

---

# TRD 测试需求分析文档
## 1. 业务流程图
## 2. 业务规则
## 3. 业务依赖关系

# STD 系统测试分析文档
## 1. 技术实现
## 2. 时序图
## 3. 实现细节

## 附：规则合并说明
## 附：待确认问题
```

## 原子事实拆解

先把源内容拆成原子事实：

- 入口：页面、菜单、任务、接口、事件、定时任务。
- 角色：用户角色、权限、系统、上游域、下游域。
- 对象：订单、发票、员工、班次、园区、配送单、配置项。
- 动作：新建、编辑、查询、审批、取消、导入、导出、同步。
- 条件：状态、时间、阈值、枚举、灰度开关、权限、上游值。
- 分支：成功、失败、空值、重复、超时、降级、部分成功。
- 结果：页面展示、状态变化、记录生成、提示信息、下游事件。
- 依赖：服务、数据表、配置中心、外部系统、用户权限。
- 失败语义：阻断、重试、忽略、降级、补偿、保留原值。
- 数据规则：必填、唯一、格式、范围、边界、精度。

## 规则标准化

- 使用“条件 -> 动作 -> 结果”表达单条规则。
- 多条件组合优先拆成按场景命名的条件块；仅在必须横向比较多个相同字段时使用决策表。
- 使用状态迁移表达有生命周期的业务对象。
- 使用 Mermaid 流程图或时序图表达多分支流程。
- 合并重复规则，但保留最严格的约束。
- 为规则保留来源章节或编号，形成追溯链。

## 业务流程图展开规则

流程图的目标是让读者沿任一分支都能回答“下一步具体做什么、到哪里、看到什么”，而不是只展示抽象阶段。

### 拆图依据

- 不同入口、角色、模式或对象类型会产生不同路由结果时，分别绘制流程图。
- 只有入口不同但后续步骤完全一致时才合并入口。
- 为清晰起见，允许在不同图中重复公共步骤，不为了去重制造跨图跳转或超大总图。

### 节点粒度

- 一个节点只表达一个动作、判断或可观察结果。
- 动作节点使用“动词 + 对象”，例如“加载匹配结果”“创建记录”“跳转目标页”。
- 判断节点必须是可回答的是/否或有限枚举问题，不使用“判断规则”“处理条件”等空泛名称。
- 路由节点写明具体目标页面、状态或业务对象，不写“进入对应页面”。
- 后置效果单独成节点，例如提示文案、结果置顶、列表定位、回流来源页、刷新数据或状态变更。

### 分支要求

- 判断节点的每条边写出明确条件或选择值。
- 每条边的下一节点写明立即执行的具体动作。
- 不把多个不同结果汇总为“处理结果”；若结果不同则继续拆分。
- 不让流程结束在中间动作上，终点前必须出现用户或系统可观察的结果。

### 绘制步骤

1. 列出入口上下文。
2. 列出公共前置步骤。
3. 找出产生路线差异的判断点。
4. 为每个分支写出立即动作、目标和后置结果。
5. 将分支与业务规则逐项对照，确认没有缺边、跨模式或错误回流。
6. 从每个入口沿每条路径人工走读到终点，确认不存在含糊节点。

## 高密度配置与枚举保真

不得为了简洁丢失源文档中可直接测试的数据。处理大表、精确内容、静态映射和条件矩阵时执行以下步骤：

1. 记录源表名称、表头、有效数据行数和关键维度。
2. 将每一行拆为条件、动作和结果；完全重复的行才允许合并。
3. 合并时给出覆盖的原始组合，确保各条件维度、时间节点、参与对象和结果差异仍可还原。
4. 数量较大的静态映射按业务对象分组列出；只有在横向比对确实更清楚时才使用附表。
5. 输出后复核枚举与组合数量，任何可测试记录都不得无说明减少。

若映射值属于地址、精确正文、提示语或枚举，应原样保留；不得改写地址、数字、比较符、标点或变量名。

## 多维交互与输出规则

对通知、审批、分流、展示、导入导出或其他多维交互，按实际存在的维度提取：

| 字段 | 要求 |
|---|---|
| 适用对象 | 角色、状态、范围或其他适用条件 |
| 触发 | 事件或时间条件、状态条件、边界日期 |
| 参与对象 | 保留源需求中的明确角色名称，不用上位概念替代 |
| 渠道/载体 | 页面、消息、文件或外部系统；未说明则写待确认 |
| 内容 | 对应模板、字段、文案或输出结构 |
| 变量 | 变量名、取值来源、计算公式和空值语义 |
| 动作 | 操作文字、目标匹配规则和执行结果 |
| 展示 | 标题、正文、颜色、图标、显隐等可观察结果 |

维度较多时，按源需求的自然业务顺序分层展开；只有文字会造成重复或难以比对时才使用决策表。必须保留改变业务含义的精确文案、提示、变量和结果差异。

## 表格使用门槛

- 默认不使用表格表达业务规则。
- 名词、条件、动作、结果、依赖、精确内容和待确认项优先使用标题、编号与列表。
- 只有同时满足“字段固定”“记录重复”“读者需要横向比较”三个条件时才使用表格。
- 即使源 PRD 使用表格，也应先判断是否可转成更易读的场景块；转换后必须保持每一原始行可追溯。
- 完成前统计 Markdown 表格数量，逐张说明其不可替代性；无充分理由则改成列表。

## 业务规则最低颗粒度

每个业务场景至少写清：

1. 适用对象和入口。
2. 触发时机。
3. 全部前置条件。
4. 正向动作与可观察结果。
5. 每个反向分支及“不操作”的明确条件。
6. 创建、更新或输出涉及的字段与计算公式。
7. 参与对象、精确内容、变量、目标或展示差异。
8. 边界值、空值、重复、状态冲突和依赖缺失。

流程图中的每个判断节点必须在业务规则正文中有对应说明，正文中的每个关键分支也必须能在流程图中找到。不得用含义模糊的“是否符合条件”替代原文能够明确指出的配置、状态或时间条件。

## 业务规则层级模板

根据需求自身的用户操作或业务能力自然归并三级标题，不限定数量。例如：

```markdown
## 2. 业务规则
### 2.1 <业务主题A>
- **规则标签**：...
- **组合规则**：
  - **条件一**：...
  - **条件二**：...

### 2.2 <业务主题B>
- **触发条件**：...
- **处理分支**：...
```

层级约束：

- 三级标题表示读者可以独立理解和测试的业务主题。
- 规则标签使用加粗短语，正文紧跟条件、动作和结果。
- 同一主题下的关联字段、按钮、反向分支放在同一规则块，不拆成多个碎片标题。
- 四级标题只在一个主题内部确有多个独立流程时使用；若内容可用加粗标签表达则不新增标题。
- TODO 紧跟产生疑点的规则块，不集中堆放。

## 业务规则组织方法

- 按业务机制和决策关系归并主题，不按页面上每个控件逐项建立章节。
- 同一主题内先写公共机制，再写限制、展示和反向分支。
- 当一组规则主要描述跳转、状态迁移或分流时，按入口上下文或模式增加四级标题。
- 每个选择或触发项使用固定结构：
  1. **触发/选择**：发生什么。
  2. **动作**：系统立即执行什么。
  3. **判断**：依据哪些上下文决定分支。
  4. **结果**：进入哪个明确页面、状态或业务对象。
  5. **表现**：提示、排序、定位、回流、显隐或其他可观察结果。
- 匹配、展示或计算类规则使用“触发机制 → 分类/计算规则 → 数量或边界 → 展示内容 → 空结果”的顺序。
- 不为凑结构重复同一事实；公共规则只写一次，差异规则按上下文展开。

### 流程与规则双向校验

- 流程图中的每个判断分支必须能在业务规则中找到对应条件、动作和结果。
- 业务规则中的每个路由、回流、提示、置顶、定位和状态变化必须能在流程图中找到节点。
- 若某项只有规则没有流程节点，判断它是否属于流程后置结果；属于则补节点，不属于则说明它是静态展示或数据约束。
- 若某个流程节点无法写成可测试规则，说明节点仍过于抽象，应继续拆解。

## 截图与原型证据

- DOCX、PDF 或网页需求包含截图时，渲染并检查相关页面，不只做文本提取。
- 只记录肉眼可观察的 UI 事实，例如卡片颜色、标题、按钮文字、字段顺序和跳转入口。
- 在规则中标注“来源：PRD 截图/原型”，区分正文规则与视觉证据。
- 不从截图猜测模板 code、接口、组件名、选择器或实现框架。

## 空白与不完整内容

以下情况必须生成就近 TODO：

- 表格只有表头或关键单元格为空。
- 章节只有标题没有正文。
- 正文引用配置表、附件或 TAB，但输入材料未包含。
- 条件使用“其他情况”“管理者”“最新记录”等未定义概念。
- 标题、正文、表格和截图之间存在时间、变量、状态或文案冲突。

TODO 必须写清缺失位置、疑点、需要确认的内容及其对测试设计的影响。

## 业务依赖与输出

- 将依赖输入与业务输出分开描述。
- 输入说明数据/配置来源、提供内容和影响范围。
- 输出说明生成或更新的业务对象及外部可观察结果，不把业务输出虚构成具体技术接口。
- 失败、重试、幂等、补偿等语义只有原文明确时才写结论；否则统一标记待确认。

## 业务规则分类

按功能实际情况选择分类：

- 查询、筛选、列表和详情展示。
- 新建、编辑、删除、启用、停用。
- 必填、格式、唯一性、枚举、边界和精度。
- 条件展示、隐藏、禁用和自动回填。
- 导入、上传、导出和下载。
- 审批、状态迁移、取消和失效。
- 上游回调、下游同步、定时任务和消息事件。
- 权限、数据范围、灰度和功能开关。
- 异常、降级、重试、幂等和并发。

## STD 证据边界

只有下列证据存在时才写技术分析：

- 明确命名的接口、DTO、事件、数据表、配置、缓存、定时任务或中间件。
- 需求中明确写出的实现说明。
- 用户提供或仓库内存在的代码与技术文档。

证据不足时写 `未发现明确信息`，或增加待确认项。不得虚构内部服务、字段、选择器、路由、SQL 或提示文案。

用户提供的期望 TAD、黄金样例或历史分析文档默认只属于格式参考，不属于当前需求的技术证据。仅在以下任一条件成立时，才能生成 STD 调用链：

- 当前工作区可访问对应代码，并能定位类、方法和调用关系。
- 当前需求材料提供明确的接口或实现说明。
- 用户明确确认某份技术文档就是当前需求的有效实现证据。

若以上条件均不成立，STD 使用以下最小表达，不生成接口清单、Job、cron、数据库、缓存或时序图：

```markdown
# STD 系统测试分析文档
## 1. 技术实现
未获取到可验证的代码、接口文档或技术实现材料，无法分析具体技术实现。

## 2. 时序图
无法获取实际调用链，不生成时序图。

## 3. 实现细节
接口、任务调度、数据表、缓存、重试、超时、降级和幂等设计均待补充技术证据后分析。
```

## 输出位置

- 默认在原始需求文件所在目录生成 Markdown。
- 文件名从需求标题与正文核心范围共同提炼，采用“业务对象 + 核心能力”。
- 去除方括号前缀、训练/需求序号、版本号和“需求”“PRD”“方案”“文档”等非业务词。
- 当标题宽泛时，以正文明确交付范围提炼文件名。
- 不因样例文档所在目录改变输出目录。
- 同名文件存在且没有明确覆盖授权时，生成候选文件或先说明冲突。

## 质量检查

- 每条业务规则至少能生成一个测试用例。
- 每个流程分支都有明确预期结果。
- 原文明确列出的枚举、节点和状态全部保留。
- 阈值、限制、精度、日期范围、提示语、路由条件和角色条件未被改写。
- 依赖关系包含失败语义。
- 图和文字之间不存在矛盾。
- 冲突或可疑示例进入待确认项，不静默修正。
